## 1. Text Storage Mode

### Overview
Text Storage mode is a lightweight alternative to KV Cache mode. It stores memories as plain JSON text files. It is ideal for:
*   **API-based LLMs** (OpenAI, Anthropic, etc.).
*   **No-GPU environments** (CPU-only deployment).
*   **Debugging** (Human-readable storage).

### Quick Start

```python
from src.conversation_manager import create_chat_manager

# Initialize with "text" mode
manager = create_chat_manager(
    storage_mode="text",
    model_id="Qwen/Qwen3-4B",  
    chat_openai_config={
        "api_key": "your-key",
        "base_url": "https://api.openai.com/v1"
    },
    aggregator_openai_config={
        "api_key": "your-key",
        "base_url": "https://api.openai.com/v1"
    },
    memory_agent_openai_config={
        "api_key": "your-key",
        "base_url": "https://api.openai.com/v1"
    }
)

# Use exactly like the KV cache manager
manager.chat("My name is Alice.")
```

## 2. Persistence & Recovery

The system features **automatic persistence**, ensuring no memory is lost between sessions.

### How It Works
*   **Auto-Save**: Metadata is saved whenever a block becomes full or a summary is generated.
*   **Auto-Load**: On startup, the system reads `kv_data/agents_metadata.json` and restores the memory graph.
*   **Lazy Loading**: Actual data (tensors/text) is only loaded from disk when a query requires it, ensuring fast startup.

### File Structure
```
kv_data/
├── agents_metadata.json          # Global registry of all memory blocks
└── kv_cache_uuid1_timestamp.pt   # Binary tensor data (KV Mode)
```

### Resetting Memory
To start a fresh session and **delete all previous memories**, set `clean_cache_first=True`:

```python
manager = create_chat_manager(
    ...,
    clean_cache_first=True  # ⚠️ WARNING: Deletes all data in kv_data/
)
```

## 3. Model Compatibility (KV Cache Only)

### ⚠️ Critical Warning
**KV Cache files (`.pt`) are strictly coupled to the specific model architecture.**

You **cannot** reuse a cache generated by `Model A` (e.g., Qwen-4B) with `Model B` (e.g., Qwen-7B). Doing so will cause dimension mismatch errors.

### Multi-Model Support
The system automatically tracks the `model_id` for every memory block.
*   **Safe Switching**: You can switch between models in different sessions. The system will only load blocks compatible with the *current* model.
*   **Isolation**: Model A's memories are hidden from Model B, but preserved on disk.

### Best Practices
1.  **Consistent Model IDs**: Use the exact same string (e.g., "Qwen/Qwen3-4B") to access previous memories.
2.  **Choose Model with Small Params**: *Qwen3-1.7B* is enough for daily use. And *Qwen3-4B* is needed for reproduce great results.

## 4. Configuration Guide

Configuration is handled via `config.yaml` or Pydantic objects. Start from either [`config.kv.yaml`](../config.kv.yaml) or [`config.text.yaml`](../config.text.yaml), then copy the one you want to `config.yaml`.

If you are unsure what each model field means, read [Config Model Roles](CONFIG_MODELS.md) before editing the config. That page explains exactly how `memory_agent_model` and `general_model` work, and how the optional role overrides inherit from `general_model`.

If you want the meaning of the memory, router, evaluation, or logging fields, read [Config Reference](CONFIG_REFERENCE.md).

The short mapping is:

- `memory_agent_model`: the memory block model itself
- `general_model`: the default non-memory LLM for the rest of the pipeline
- `manager_model`: optional override for the top-level chat / tool-calling model
- `aggregator_model`: optional override for the model that merges retrieved memories
- `router_fallback_model`: optional override for the router LLM fallback
- `question_answer_model`: optional override for the evaluation-only final answer model

If an override is omitted or set to `null`, E-mem uses `general_model`.

### Key Parameters (`config.yaml`)

```yaml
model:
  memory_agent_model:
    model_context_window: 32768

memory:
  storage_mode: "kv_cache"      # or "text"
  max_blocks: 5                 # Max blocks to retrieve per query
  
  # Router Settings
  router_type: "hybrid"
  hybrid_router:
    embedding_provider: "huggingface"
    summary_weight: 0.3         # Global Alignment
    text_weight: 0.4            # Semantic Association
    bm25_weight: 0.3            # Keyword Matching
    bm25_boost_threshold: 0.8   # Force include block if keyword match is strong
```
For more information, see the [API Reference](API_REFERENCE.md), [Config Model Roles](CONFIG_MODELS.md), [Config Reference](CONFIG_REFERENCE.md), [KV example config](../config.kv.yaml), and [text example config](../config.text.yaml).

### Validation
All config is validated at runtime. Invalid values (e.g., weights not summing to ~1.0, invalid paths) will raise a `ConfigurationError` immediately.
